# Практический roadmap улучшений POE2 LootLens

Ниже перечислены не абстрактные «best practices», а изменения, которые дадут проекту измеримую пользу. Порядок — от наиболее полезных к более долгосрочным.

## Приоритет 1 — качество и воспроизводимость OCR

1. **Собрать локальный regression corpus реальных строк.** Хранить обезличенные кропы проблемных строк и ожидаемый результат; каждый OCR-рефакторинг прогонять по этому набору, чтобы улучшение одной награды не ломало другие.
2. **Разделить OCR на явные стадии.** `row detection → preprocessing variants → OCR candidates → normalization → market matching → temporal confirmation`; это упростит тестирование каждой стадии отдельно и уменьшит объём изменений в больших классах.
3. **Логировать выбранного и второго кандидата.** Для unmatched/fuzzy строк записывать top-2 совпадения, score и margin; это позволит понимать, проблема в OCR или в словаре, без гадания по одному raw-тексту.
4. **Добавить опциональный диагностический дамп только проблемной строки.** По явному флагу пользователя сохранять кропы unmatched-строк локально с ограничением размера/числа файлов; по умолчанию функция должна быть выключена из соображений приватности.
5. **Профили OCR по масштабу интерфейса.** В калибровке определять примерную высоту строки и выбирать масштаб/пороговые варианты по профилю, а не одинаково для 1080p, 1440p и 4K.

## Приоритет 2 — устойчивость desktop-приложения

6. **Ввести единое состояние жизненного цикла модулей.** Вместо набора `_operationInProgress`, nullable-ссылок и отдельных bool использовать состояния `Stopped/Starting/Running/Stopping/Faulted`; это устранит гонки при быстрых кликах, закрытии и смене настроек.
7. **Добавить глобальный crash handler и отчёт об ошибке.** Перехватывать UI/TaskScheduler/AppDomain exceptions, записывать последнюю операцию, версию, настройки без чувствительных данных и предлагать открыть папку отчёта.
8. **Сделать резервную копию конфигурации.** При сохранении держать `config.json.bak`; при повреждении основного файла восстанавливать backup и явно сообщать пользователю, а не молча сбрасывать всё к defaults.
9. **Не выполнять долгие остановки модулей на UI thread.** Полное закрытие сейчас синхронно ждёт OCR-модули; лучше асинхронно завершать их с коротким экраном «Завершение…» и жёстким timeout.
10. **Добавить per-monitor DPI awareness и тесты координат.** WPF, WinForms overlay и `System.Drawing` используют разные координатные модели; отдельный слой преобразования нужен для нескольких мониторов с разным масштабом.

## Приоритет 3 — настройки и UX

11. **Показывать зависимости настроек прямо в форме.** Не только MessageBox: рядом с автозапуском оценщика показывать постоянный статус `Область выбрана/не выбрана` и кнопку перехода к калибровке.
12. **Добавить страницу «Состояние системы».** Версия OCR-моделей, последняя цена/курс, возраст кеша, активные модули, выбранная область, текущая лига и последние ошибки в одном месте сильно упростят поддержку.
13. **Добавить кнопку «Экспорт диагностики».** Один ZIP с логами, config без приватных путей, версиями и метаданными экрана; без скриншотов, если пользователь отдельно их не разрешил.
14. **Сохранять позицию и размер главного окна.** Валидировать координаты при старте и возвращать окно в видимую рабочую область после смены монитора.
15. **Разделить portable и installer-сборки.** Portable остаётся текущим вариантом; installer может добавить корректный автозапуск Windows, ярлыки, uninstall и сохранение пользовательских данных вне каталога программы.

## Приоритет 4 — данные и релизы

16. **Показывать свежесть рыночных данных.** В оверлее/главном окне различать `актуальные`, `устаревший кеш`, `источник недоступен`; цена без времени последнего успешного обновления может вводить в заблуждение.
17. **Абстрагировать рыночные источники.** Единый интерфейс source adapter позволит безопасно добавлять резервный источник и тестировать 404/429/timeout без изменения PriceRepository.
18. **Автоматизировать CI на GitHub Actions.** Windows job: restore, build, tests, publish, ZIP, SHA-256 и draft prerelease; ручная локальная сборка останется, но риск забыть тест или файл уменьшится.
19. **Подписывать релизные бинарники.** Code signing снизит предупреждения SmartScreen и повысит доверие, когда проект начнёт распространяться шире.
20. **Ввести формат миграций конфигурации по версиям.** Вместо одного блока условных исправлений — последовательные миграции `v6→v7`, `v7→v8`, каждая с отдельными тестами.
